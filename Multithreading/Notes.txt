* Thread :
    -> A thread is a lightweight sub-process, the smallest unit of execution in a program.
	-> Multiple threads can run concurrently to utilize CPU efficiently.
	-> In Java, threads are managed by JVM, and every Java program has a main thread by default.

* Difference between Thread & Process :

	    Process					|		   Thread
    ----------------------------|---------------------------------------------------------
	-> Heavyweight				|    ->  Lightweight
	-> Own memory				|    ->  Shares memory with other threads in same process
    -> Inter-process            |    ->  Communication is easier
	   communication is costly	|	    (shared memory)
                                |


* Mutithreading :
	-> Multithreading means running multiple threads concurrently within a single program.
	-> Hence, Instead of running tasks sequentially [Task1 → Task2 → Task3], 
	   we can run task concurrently / parallelly [Task1 + Task2 + Task3]

* Need of Multithreading :
	-> Better CPU Utilization (As the CPU never stays idle)
	-> Faster execution (As tasks run concurrently / parallelly)

* Real World Multithreading examples :
	-> Web Servers (Multiple Users)
	-> Games
	-> Chat Apps

* Concurrency vs Parallelism
	Concurrency :
		-> Multiple tasks make progress by interleaving execution.
		-> Uses context switching
		-> Better for single core CPU / IO heavy tasks.
		-> Tasks are not truly simultaneous.
		-> Real world example : When one person is both cooking food and replying to messages.
		   The person switches between the two tasks and both tasks progress together. 
		   But the two tasks don't happen exactly at the same moment.
	Parallelism :
		-> Multiple tasks execute literally at the same time.
		-> Uses multiple CPU cores.
		-> Better for CPU heavy tasks.
		-> No need for switching between those tasks.
		-> Real world example : When one person cooks food and another person replies to the messages.
		   The two tasks are truly simultaneous.
	
	|---------------------------------------------------------------------------------------|
	| Note - The JVM creates and manages threads, but it does not decide how they run.      |
	|		 The OS scheduler determines execution - it decides which threads run, when,    |
    |        and on which CPU cores, based on factors like thread state (ready/blocked),	|
	|		 priority, and CPU availability.												|
	|		 As a result, concurrent or parallel execution of threads emerges automatically |
	|		 depending on hardware (single core vs multi core) and the scheduling.			|
	|_______________________________________________________________________________________|


* Ways of Creating Threads in java  :
	1. Extending the thread class
	2. Implementing Runnable Interface
	3. Creating threads using Lambda Expression (Java 8+)

  |--------------------------------------------------------------------------------------------------|
  | Note - Implementing Runnable interface for creating threads is better than extending.            |
  |		  the thread class.																			 |
  |		  This is because Java doesn't support multiple inheritance and we can extend another class  |
  |		  according to our needs while implementing runnable interface at the same time.             |
  |		  Modern Java always prefer runnable interface or lambda for creating threads.               |
  |__________________________________________________________________________________________________|


* Thread Life Cycle
	-> Thread life cycle in java describes the different states a thread goes through from creation to termination.
	-> In java, this lifecycle is defined in the Thread.State enum.

	1. NEW (Created) : 
		Thread is created but not started yet. start() method has not been called. 
		STATE = NEW.

		|--------------------------|
		| Thread t = new Thread(); |
		|__________________________|
	
	2. RUNNABLE (Ready/Running) :
		After calling start, thread moves to RUNNABLE. It is ready to run and waiting for CPU.
		Java combines ready + running into one state.
		STATE = RUNNABLE.

		|-----------|
		| t.start() |
        |___________|
	
	3. BLOCKED :
		Thread is waiting to acquire a lock (monitor).
		Happens in synchronized blocks.
		STATE = BLOCKED.

		|--------------------------------------------------------------------|
		| synchronized(obj) {											     |		
    	|	 // if another thread holds lock, current thread becomes BLOCKED |
		| } 																 |	
        |____________________________________________________________________|
	
	4. WAITING :
		Thread waits indefinitely for another thread to perform an action.
		Common methods - wait(), join()
		STATE = WAITING

		|-----------|
		| t.wait();	|										     		
    	| t.join(); |
		|___________|  	

	5. TIMED WAITING :
		Thread waits for a specific amount of time.
		Common methods - sleep(time), wait(time), join(time). 
		Time is specified in ms.
		STATE = TIMED WAITING 	

		|---------------------|
		| Thread.sleep(1000); |
        |_____________________|

	6. TERMINATED (Dead) :
		Thread has finished execution or stopped.
		STATE = TERMINATED


	# Lifecycle	flow :
		NEW → RUNNABLE → (BLOCKED / WAITING / TIMED WAITING) → RUNNABLE → TERMINATED

	# Important methods :
		start() - Starts thread
		run() - Thread logic
		sleep(ms) - Pause thread
		join() - Wait for another thread
		yield() - Suggest CPU switch



* Thread Priority :
	-> Thread priority is a hint to the OS thread scheduler about which thread should get
	   CPU time first.
	-> Each thread has a priority value. Higher priority means more chance of early execution
	   and lower priority means less chance.
	-> Java thread priority is not a guarantee, only a suggestion to the OS thread scheduler.
	-> Thread priority in java ranges from 1 to 10 (default priority being 5).
	-> Java provides 3 constant priorities 
		1. Thread.MIN_PRIORITY = 1
		2. Thread.NORM_PRIORITY = 5 (Default)
		3. Thread.MAX_PRIORITY = 10
	-> Java uses the underlying OS thread scheduler (not purely JVM controlled).
	   Scheduler may consider
	    -  Thread priority
		-  CPU availability
	    -  Thread state (Runnable, Waiting etc.)
        -  OS scheduling algorithm (Round Robin etc.)
	   So, final decision of thread scheduling depends primarily on the OS, with JVM providing hints (like priority).
	   Hence, behaviour of thread priority may differ on different operating systems (Windows, Linux etc.)
