# Thread :
    -> A thread is a lightweight sub-process, the smallest unit of execution in a program.
	-> Multiple threads can run concurrently to utilize CPU efficiently.
	-> In Java, threads are managed by JVM, and every Java program has a main thread by default.

# Difference between Thread & Process :

	    Process					|		   Thread
    ----------------------------|---------------------------------------------------------
	-> Heavyweight				|    ->  Lightweight
	-> Own memory				|    ->  Shares memory with other threads in same process
    -> Inter-process            |    ->  Communication is easier
	   communication is costly	|	    (shared memory)
                                |


# Mutithreading :
	-> Multithreading means running multiple threads concurrently within a single program.
	-> Hence, Instead of running tasks sequentially [Task1 → Task2 → Task3], 
	   we can run task concurrently / parallelly [Task1 + Task2 + Task3]

# Need of Multithreading :
	-> Better CPU Utilization (As the CPU never stays idle)
	-> Faster execution (As tasks run concurrently / parallelly)

# Real World Multithreading examples :
	-> Web Servers (Multiple Users)
	-> Games
	-> Chat Apps

# Concurrency vs Parallelism
	Concurrency :
		-> Multiple tasks make progress by interleaving execution.
		-> Uses context switching
		-> Better for single core CPU / IO heavy tasks.
		-> Tasks are not truly simultaneous.
		-> Real world example : When one person is both cooking food and replying to messages.
		   The person switches between the two tasks and both tasks progress together. 
		   But the two tasks don't happen exactly at the same moment.
	Parallelism :
		-> Multiple tasks execute literally at the same time.
		-> Uses multiple CPU cores.
		-> Better for CPU heavy tasks.
		-> No need for switching between those tasks.
		-> Real world example : When one person cooks food and another person replies to the messages.
		   The two tasks are truly simultaneous.
	
	|---------------------------------------------------------------------------------------|
	| Note - The JVM creates and manages threads, but it does not decide how they run.      |
	|		 The OS scheduler determines execution - it decides which threads run, when,    |
    |        and on which CPU cores, based on factors like thread state (ready/blocked),	|
	|		 priority, and CPU availability.												|
	|		 As a result, concurrent or parallel execution of threads emerges automatically |
	|		 depending on hardware (single core vs multi core) and the scheduling.			|
	|_______________________________________________________________________________________|

# Thread Life Cycle
	-> Thread life cycle in java describes the different states a thread goes through from creation to termination.
	-> In java, this lifecycle is defined in the Thread.State enum.

	1. NEW (Created) : 
		Thread is created but not started yet. start() method has not been called. 
		STATE = NEW.

		|--------------------------|
		| Thread t = new Thread(); |
		|__________________________|
	
	2. RUNNABLE (Ready/Running) :
		After calling start, thread moves to RUNNABLE. It is ready to run and waiting for CPU.
		Java combines ready + running into one state.
		STATE = RUNNABLE.

		|-----------|
		| t.start() |
        |___________|
	
	3. BLOCKED :
		Thread is waiting to acquire a lock (monitor).
		Happens in synchronized blocks.
		STATE = BLOCKED.

		|--------------------------------------------------------------------|
		| synchronized(obj) {											     |		
    	|	 // if another thread holds lock, current thread becomes BLOCKED |
		| } 																 |	
        |____________________________________________________________________|
	
	4. WAITING :
		Thread waits indefinitely for another thread to perform an action.
		Common methods - wait(), join()
		STATE = WAITING

		|-----------|
		| t.wait();	|										     		
    	| t.join(); |
		|___________|  	

	5. TIMED WAITING :
		Thread waits for a specific amount of time.
		Common methods - sleep(time), wait(time), join(time). 
		Time is specified in ms.
		STATE = TIMED WAITING 	

		|---------------------|
		| Thread.sleep(1000); |
        |_____________________|

	6. TERMINATED (Dead) :
		Thread has finished execution or stopped.
		STATE = TERMINATED


	* Lifecycle	flow :
		NEW → RUNNABLE → (BLOCKED / WAITING / TIMED WAITING) → RUNNABLE → TERMINATED

	* Important methods :
		start() - Starts thread
		run() - Thread logic
		sleep(ms) - Pause thread
		join() - Wait for another thread
		yield() - Suggest CPU switch
	
	* start() can be called only once for a thread object. 
	  This is because start() can only be called when the thread is in NEW state.
	  If you call start() more than once on the same Thread object, java will throw
	  IllegalThreadStateException (Runtime Exception).


# Thread Priority :
	-> Thread priority is a hint to the OS thread scheduler about which thread should get
	   CPU time first.
	-> Each thread has a priority value. Higher priority means more chance of early execution
	   and lower priority means less chance.
	-> Java thread priority is not a guarantee, only a suggestion to the OS thread scheduler.
	-> Thread priority in java ranges from 1 to 10 (default priority being 5).
	-> Java provides 3 constant priorities 
		1. Thread.MIN_PRIORITY = 1
		2. Thread.NORM_PRIORITY = 5 (Default)
		3. Thread.MAX_PRIORITY = 10
	-> Java uses the underlying OS thread scheduler (not purely JVM controlled).
	   Scheduler may consider
	    -  Thread priority
		-  CPU availability
	    -  Thread state (Runnable, Waiting etc.)
        -  OS scheduling algorithm (Round Robin etc.)
	   So, final decision of thread scheduling depends primarily on the OS, with JVM providing hints (like priority).
	   Hence, behaviour of thread priority may differ on different operating systems (Windows, Linux etc.)

# Race Condition :
	-> A race condition in Java threads occurs when multiple threads access 
       and modify shared data concurrently, and the final result depends on 
       the timing or order of execution rather than the intended logic.
	-> Simply put, two or more threads are “racing” to use or modify the same resource.
       Since thread scheduling is unpredictable, the outcome can be inconsistent or incorrect.
	-> This happens because threads share the same heap memory, and operations like 
       count++ are not atomic. They involve multiple steps (read → modify → write).
       Due to unpredictable thread scheduling, a race condition may occur.
	-> Example:
       Two threads, Thread A and Thread B, access the shared variable `count` 
       at the same time. Both threads read the value as 0. Each thread then 
       increments the value to 1 independently. After that, both threads write 
       the updated value back to memory.

	   Since both write operations store the value 1, the final result becomes 1 
       instead of the expected 2.

	   Hence, the final result differs from what the logic intended.
       This situation is called a race condition.

# Thread Synchronization :
	-> In Java, multiple threads can run concurrently, but if they access the same data 
	   without control, it can lead to inconsistent results.
	-> Thread synchronization is the process of controlling access to shared resources 
	   by multiple threads to avoid race conditions and ensure data consistency.
	-> Synchronization ensures that only one thread can access the critical section at a time
	   and it prevents multiple threads from entering a critical section simultaneously.
	   Think of it as a traffic signal for threads.
	
	* Synchronized keyword : 
	  Java provides the synchronized keyword to handle thread synchronization.
	  In Java, every object has an intrinsic lock (monitor).
	  'synchronized' blocks/methods acquire this lock before executing code.
	  While a thread holds the lock, no other thread can enter another 'synchronized'
	  block/method on the same object.
	  
	  1. Synchronized method - A method can be declared synchronized. This ensures that 
	     only one thread can execute the method on an object at a time.
	  2. Synchronized block - Sometimes, you don’t want to lock the entire method, 
	     only a part of the code. That’s when synchronized blocks are useful.


# Inter-Thread Communication :
	-> Inter-thread communication is a mechanism by which multiple threads communicate 
	   and coordinate with each other to avoid problems like busy waiting, race conditions, 
	   and data inconsistency.

       |-------------------------------------------------------------------------------------|
	   |  * Busy Waiting (also called spinning) :                                            |
	   |  - A situation when a thread keeps checking a condition repeatedly in a loop.       |
	   |  - Problems : Wastes CPU, Reduces Performance, and is inefficient.                  |
       |                                                                                     |
	   |  * Data Inconsistency :                                                             |
	   |  - Happens when multiple threads access and modify shared data concurrently without |  
	   |    proper synchronization.                                                          |
       |  - Problems : Leads to incorrect or unpredictable results.                          |
       |                                                                                     |
	   |  * Race Condition (As explained earlier) :                                          |
	   |  - Occurs when two or more threads access shared data concurrently and the final    |
	   |    outcome depends on the timing of their execution.                                |
	   |  - Closely related to data inconsistency.                                           |
       |_____________________________________________________________________________________|


	-> Inter-Thread Communication is - "Efficient waiting & coordination between threads"
	-> Key Methods (from Object Class) that helps in Inter-Thread Communication :

	   |-----------------------------------------------------------------------------------|	
       |  * wait() :                                                                       |
	   |  - Causes current thread to release the lock & go into WAITING state.             |
	   |  - It waits until another thread calls notify() or notifyAll().                   |
	   |                                                                                   |
       |  * notify() :                                									   |	
	   |  - Wakes up one waiting thread.												   |	
	   |  - That thread will compete to acquire the lock again.							   |
	   |																				   |
	   |  * notifyAll() :																   |
	   |  - Wakes up all waiting threads.												   |	
	   |  - Useful when multiple threads are waiting.  									   |
       |___________________________________________________________________________________|
	   
	   * Important Points for the above methods :
	   - These methods must be called inside synchronized blocks/methods.
	   - These methods must be called on the same object whose lock is used.
	   - If 3 threads (T1, T2, T3) are waiting on the same object’s monitor and another 
	     thread (T4) calls notify(), only one thread would be woken up, but knowing which 
		 thread would be woken is unpredictable and depends on the JVM scheduler, while 
		 the remaining threads keep waiting.

	-> Lifecycle flow when 2 threads are performing inter-thread communication :
		
		|------------------------------------------------|
		| Thread enters synchronized block				 |
		| Calls wait() → releases lock → goes to WAITING |
		| Another thread calls notify()					 |
		| Waiting thread moves to BLOCKED				 |
		| When lock is free → resumes execution			 |	 
	    |________________________________________________|
		
	
# Sleep() method :
	-> Method of 'Thread' class.
	-> Can be called anywhere; does not require 'synchronized' block/method.
	-> Unlike wait(), sleep() does not release any lock held by the thread.
	-> Used to pause execution for a fixed time.
	-> Throws Checked Exception (InterruptedException).

# Yield() method :
	-> yield() is a static method of the 'Thread' class.
	-> Does not require 'synchronized' block/method to execute.
	-> It is used to pause the currently executing thread temporarily and give other 
	   threads a chance to execute.
	-> The current thread voluntarily moves from RUNNING to RUNNABLE state.
	-> It hints the scheduler that it is willing to give up CPU so that other runnable 
	   threads of the same priority can execute, but it is not guaranteed to 
	   happen.
	-> Even if that happens, how long the CPU will be taken by another thread is entirely 
	   up to the scheduler.
	-> Unlike wait(), yield() does not interact with any object's monitor.
